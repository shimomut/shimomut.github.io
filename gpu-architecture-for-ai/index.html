1. Why GPUs for AI

Modern AI is almost entirely dense linear algebra — huge matrix multiplies, element-wise operations, and reductions applied to tensors with millions of elements. These operations are embarrassingly parallel: each output element depends on an independent dot product. CPUs devote most of their silicon to control logic and caches for a small number of complex threads. GPUs flip the ratio: tens of thousands of simple arithmetic lanes, minimal per-thread control state, and a huge memory bandwidth pipe to feed them.

The numbers tell the story. A high-end server CPU delivers roughly 1-3 TFLOPS of FP32 compute. An H100 GPU delivers ~2,000 TFLOPS of BF16 Tensor Core compute — three orders of magnitude more — by organizing its silicon as 132 Streaming Multiprocessors that each run thousands of threads in parallel.

2. The Execution Hierarchy

CUDA exposes four nested levels of parallelism. Understanding this hierarchy is the key to everything else.

  • Thread — the smallest unit. Each thread executes the same function (called a kernel) but on different data, identified by its index.
  • Warp — a group of 32 threads that execute in lockstep. This is the hardware's actual scheduling unit: the SM issues one instruction to all 32 lanes per cycle.
  • Block (also thread block or CTA) — a group of warps (typically 128-1024 threads) that run together on one SM. Threads in a block can share memory and synchronize with each other.
  • Grid — the set of all blocks launched by a single kernel call. Blocks within a grid are independent and can be scheduled on any SM in any order.

3. The Streaming Multiprocessor (SM)

The SM is the GPU's fundamental unit of execution — the equivalent of a CPU core, but vastly wider. An H100 has 132 SMs. Each SM packs:

  • 4 sub-partitions, each with its own warp scheduler, register file, and set of execution units.
  • 128 CUDA Cores for scalar FP32/INT32 math (32 per sub-partition).
  • 4 Tensor Cores for matrix-multiply-accumulate (one per sub-partition).
  • 256 KB register file and 228 KB unified shared memory / L1.
  • A warp scheduler per sub-partition that picks one ready warp per cycle to issue an instruction.

4. Warps and SIMT Execution

NVIDIA calls its execution model SIMT — Single Instruction, Multiple Threads. All 32 threads in a warp share a program counter. On each cycle, the warp scheduler picks a ready warp and broadcasts one instruction to all 32 lanes. Each lane operates on its own data, but every lane runs the same instruction at the same time.

This is efficient when all threads follow the same control flow. When an if makes threads disagree — called warp divergence — the warp executes both branches serially, masking off the threads not taking each path. Divergence can halve (or worse) the effective throughput of a warp.

5. Memory Hierarchy at a Glance

A GPU's memory system is a pyramid: small and fast at the top, huge and slow at the bottom. AI kernels spend most of their effort arranging data to maximize reuse of the fast tiers.

6. Tensor Cores

CUDA Cores do one FP32 multiply-add per cycle per lane — fine for general code, slow for AI. Tensor Cores are specialized units that perform an entire small matrix-multiply-accumulate (MMA) in a single instruction: something like D = A · B + C where A, B, C, D are 16×16 tiles.

A single warp can issue one Tensor Core MMA that performs thousands of FMAs in a handful of cycles. This is why an H100 achieves ~2 PFLOPS of BF16 compute while having "only" 16,896 CUDA Cores: the Tensor Cores provide an additional 16× boost on top.

7. CUDA Programming Model & Libraries

CUDA is NVIDIA's C++-based programming model for GPUs. You write a kernel — a function tagged __global__ — and launch it with a triple-angle-bracket syntax that specifies the grid and block dimensions: 

__global__ void add(const float* a, const float* b, float* c, int n) {
    int tid = blockIdx.x * blockDim.x + threadIdx.x;  // global thread index
    if (tid < n) c[tid] = a[tid] + b[tid];
}

int main() {
    int n = 1<<20;
    dim3 block(256);                      // 256 threads per block (8 warps)
    dim3 grid((n + 255) / 256);           // enough blocks to cover n elements
    add<<<grid, block>>>(a, b, c, n);     // kernel launch
}

Writing kernels by hand is rare in production AI. The real power of CUDA is its library stack. Most high-level code (PyTorch, JAX, TensorFlow) ultimately calls into these libraries for the heavy lifting:

cuBLAS

Dense linear algebra (GEMM, GEMV). The workhorse behind torch.matmul. Tuned per shape for each GPU generation.

cuDNN

Deep-learning primitives: convolution, pooling, batch-norm, RNN cells, attention. Handles the gnarly algorithm selection and kernel fusion.

CUTLASS

A C++ template library of GEMM building blocks. Used to write custom, shape-specialized kernels (e.g., Flash Attention, mixture-of-experts).

NCCL

Collective communication across GPUs: all-reduce, all-gather, broadcast. The substrate of multi-GPU and multi-node training.

Thrust / CUB

Parallel algorithm primitives: scan, sort, reduction. Building blocks for custom kernels.

TensorRT

Inference optimizer and runtime: fuses ops, picks kernels, quantizes, and builds a deployment-ready engine from a trained model.

CUDA Graphs

Capture a sequence of kernel launches once, replay it many times with near-zero CPU overhead. Critical for low-latency inference.

Triton

A Python DSL (developed at OpenAI) that compiles to PTX and rivals hand-written CUTLASS kernels for many AI workloads.

8. Putting It Together — How a Kernel Runs

When you launch a kernel with, say, 4096 blocks on a GPU with 132 SMs, the hardware's Gigathread Engine dispatches blocks to SMs as space becomes available. Each SM runs multiple blocks concurrently (limited by registers and shared memory, as the occupancy calculator above showed). When a block finishes, the SM is immediately given the next queued block.

Summary

The whole pipeline for an AI operation:

  1. PyTorch (or JAX, TF) calls a cuBLAS / cuDNN / custom kernel.
  2. The kernel is launched as a grid of blocks.
  3. The Gigathread Engine dispatches blocks to the 132 SMs.
  4. Each SM breaks its blocks into warps of 32 threads.
  5. Warp schedulers pick ready warps each cycle and issue instructions on the CUDA Cores (scalar math) or Tensor Cores (tile MMAs).
  6. Data flows up the memory hierarchy from HBM → L2 → shared memory → registers, and results flow back down.
