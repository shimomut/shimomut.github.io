LLM Inference Performance Calculator

Estimate TTFT, throughput, and find the bottleneck for any model / GPU / topology

What this calculates

LLM inference has two very different phases. Prefill (processing the input prompt) is a dense matrix-matrix operation and is usually compute-bound. Decode (generating tokens one at a time) is memory-bandwidth-bound because every token must re-read the entire weight matrix through HBM.

The calculator below uses first-principles estimates from three numbers per GPU (FP16/BF16 TFLOPS, HBM bandwidth, and interconnect bandwidth) plus the model's parameter count and architecture. Pick a model, a GPU, a topology, and see where the time goes.

Rule-of-thumb model used here:
  • Prefill per layer = max(compute-time, hbm-time) + all-reduce-time.
  • Decode per layer = max(weight-read-time, kv-read-time, compute-time) + all-reduce-time.
  • TTFT = prefill-time-for-prompt. Throughput = batch / decode-time-per-token.

The math behind it

Using P = active params, L = layers, H = hidden, S = sequence length, B = batch, N = TP degree, b = bytes per weight, KV in FP16 (2 B):

Prefill (TTFT)

  • Linear FLOPs: 2 · P · B · S
  • Attention FLOPs (quadratic in S): 4 · L · B · S² · H
  • Weight bytes read: P · b (once, reused across B·S tokens)
  • All-reduce per layer: 2 × B · S · H · 2 bytes (activations in FP16)
  • Per-layer time ≈ max(compute, hbm) + allreduce

Decode (per token)

  • Weight bytes read: P · b / N per GPU
  • KV bytes read: 2 · L · S · Hkv · B · 2 (grows with context)
  • Linear FLOPs: 2 · P · B
  • Attention FLOPs: 4 · L · B · S · H
  • All-reduce per layer: 2 × B · H · 2 bytes (only 1 token)

Tensor parallel collective

Ring all-reduce takes 2 · (N − 1) / N · bytes / BW. The slowest link on the ring sets BW: if all N GPUs fit in one node, BW is NVLink; if the ring crosses nodes, BW drops to the inter-node link (InfiniBand).

What this skips

  • Kernel launch overhead, scheduler cost, Python overhead. Real systems add 10-50 µs per step.
  • FlashAttention & PagedAttention details — we assume a reasonably optimized implementation.
  • Pipeline parallelism and expert parallelism for MoE (we approximate with active-param scaling).
  • KV-cache quantization (assumes FP16 KV).
