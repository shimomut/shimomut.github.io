1. Self-Attention

Concept

Self-attention is the fundamental operation that gives Transformers their power. For each token in a sequence, self-attention computes a weighted sum of all tokens' representations — allowing every word to directly "look at" every other word, regardless of distance. This is what lets a Transformer understand that in "The cat sat on the mat because it was tired", the word "it" refers to "cat" and not "mat".

Each input token is projected into three vectors: a Query (Q) — "what am I looking for?", a Key (K) — "what do I contain?", and a Value (V) — "what information do I provide?". The attention score between two tokens is the dot product of one token's Query with another's Key. These scores are scaled and passed through softmax to produce attention weights, which are then used to compute a weighted sum of the Value vectors.

Attention(Q, K, V) = softmax(Q · KT / √dk) · V

The division by √dk (the dimension of the key vectors) prevents the dot products from growing too large, which would push softmax into regions with extremely small gradients.

Key Points

  • Global context: Every token can attend to every other token — no recurrence or convolution needed.
  • Parallelizable: All attention scores can be computed simultaneously (unlike RNNs which process sequentially).
  • Quadratic cost: Computing all pairwise scores costs O(n²) in sequence length — the main scalability challenge.
  • Position-agnostic: Self-attention itself has no notion of order; positional encodings must be added separately.

2. The KV Cache

Concept

During autoregressive generation (producing one token at a time), the model computes attention for each new token against all previous tokens. Without optimization, this means recomputing the Key and Value projections for every past token at every generation step — an enormous waste of computation.

The KV Cache solves this by storing the Key and Value vectors from all previous tokens. When generating token t, only the new token's Q, K, and V need to be computed. The new K and V are appended to the cache, and the Query attends to the full cached K and V. This turns generation from O(n²) total compute to O(n) per step.

The trade-off is memory: the KV cache grows linearly with sequence length and is proportional to 2 × num_layers × num_heads × head_dim × seq_len × bytes_per_element. For a 70B model with 128K context, the KV cache alone can consume tens of gigabytes per request — making it the dominant memory bottleneck in LLM serving. This is exactly why optimizations like Multi-Query Attention and Grouped-Query Attention exist.

Key Points

  • Avoids recomputation: Past tokens' K and V are computed once and reused for all future steps.
  • Memory-bound: The KV cache is the primary memory consumer during inference — it scales with sequence length, batch size, and model depth.
  • Motivates MQA/GQA: Reducing the number of KV heads directly shrinks the cache, enabling longer contexts and larger batches.
  • PagedAttention: Systems like vLLM manage KV cache memory in pages (like OS virtual memory) to eliminate fragmentation and enable efficient batching.

3. Multi-Head Attention (MHA)

Concept

A single attention function can only capture one type of relationship at a time. Multi-Head Attention runs multiple attention operations in parallel — each "head" uses different learned projection matrices (WQ, WK, WV), allowing the model to simultaneously attend to information from different representation subspaces at different positions.

For example, in the sentence "The cat sat on the mat because it was tired", one head might learn to track syntactic relationships (subject-verb), another might capture coreference ("it" → "cat"), and another might focus on positional proximity. The outputs of all heads are concatenated and projected through a final linear layer.

MultiHead(Q, K, V) = Concat(head1, ..., headh) · WO
where headi = Attention(Q·WiQ, K·WiK, V·WiV)

The total computation is the same as a single head with full dimensionality — each head operates on dmodel/h dimensions, so the cost is equivalent. The benefit is purely representational: multiple independent attention patterns are more expressive than a single one.

Key Points

  • Multiple perspectives: Each head can learn a different type of relationship (syntactic, semantic, positional).
  • Same total cost: h heads of dimension d/h costs the same as 1 head of dimension d.
  • Standard in all Transformers: GPT, BERT, LLaMA, etc. all use multi-head attention — typically 32, 64, or 128 heads.
  • KV cache cost: Each head maintains its own K and V cache — this is the memory problem MQA and GQA address.

4. Multi-Query Attention (MQA)

Concept

Multi-Query Attention, introduced by Noam Shazeer in 2019, is a simple but powerful optimization: instead of giving each attention head its own Key and Value projections, all heads share a single set of Keys and Values. Each head still has its own unique Query projection, so heads can still attend to different aspects of the input — but they all read from the same K and V.

The impact on inference is dramatic. For a model with 64 attention heads, MQA reduces the KV cache size by 64×. This means you can serve longer sequences, fit more concurrent requests in GPU memory, and the memory bandwidth bottleneck during the decode phase (which is dominated by reading the KV cache) is greatly reduced.

The trade-off is a small quality degradation — with only one set of K/V, the model has less representational capacity. In practice, the quality loss is modest and often acceptable for inference-optimized models.

⚡ How It Differs from Standard MHA

  • MHA: h Query heads, h Key heads, h Value heads → KV cache = h × dhead × seq_len × 2
  • MQA: h Query heads, 1 Key head, 1 Value head → KV cache = 1 × dhead × seq_len × 2
  • KV cache memory reduced by a factor of h (e.g., 64× for a 64-head model).
  • Decode throughput improves because less memory bandwidth is needed to read the KV cache.
  • Small quality trade-off — mitigated by GQA (next section).

Key Points

  • Inference optimization: MQA is primarily about making inference faster and more memory-efficient.
  • Used by: PaLM, Falcon, StarCoder, and other inference-optimized models.
  • Bandwidth-bound decode: The decode phase reads the entire KV cache for every token — shrinking it directly improves throughput.
  • Can be retrofitted: An MHA model can be converted to MQA by averaging or selecting one set of KV heads, then fine-tuning briefly.

5. Grouped-Query Attention (GQA)

Concept

Grouped-Query Attention is the middle ground between MHA and MQA. Instead of every head having its own KV (MHA) or all heads sharing one KV (MQA), GQA divides the query heads into groups, and each group shares one set of Key and Value heads.

For example, with 32 query heads and 8 KV groups, every 4 query heads share the same K and V. This gives a 4× reduction in KV cache size (compared to MHA) while retaining much more representational capacity than MQA's single KV head.

GQA has become the de facto standard for modern LLMs. LLaMA 2 (70B), LLaMA 3, Mistral, and Gemma all use GQA. It achieves nearly the same quality as MHA while providing most of the inference efficiency benefits of MQA.

⚡ How It Differs

  • MHA: h Q heads, h KV heads → full KV cache
  • MQA: h Q heads, 1 KV head → minimal KV cache, some quality loss
  • GQA: h Q heads, g KV heads (1 < g < h) → balanced KV cache, near-MHA quality
  • GQA with g=1 is equivalent to MQA; GQA with g=h is equivalent to MHA.
  • Typical configurations: 32 Q heads with 8 KV groups (LLaMA 2 70B), or 8 Q heads with 4 KV groups.

Key Points

  • Best of both worlds: Near-MHA quality with significant KV cache savings.
  • Industry standard: LLaMA 2/3, Mistral, Gemma, CodeLlama all use GQA.
  • Tunable trade-off: The number of KV groups is a hyperparameter — more groups = better quality but larger cache.
  • Uptraining: Existing MHA models can be converted to GQA by grouping and averaging KV heads, then fine-tuning for a small fraction of the original training compute.

6. Flash Attention

Concept

Flash Attention (Dao et al., 2022) is an IO-aware exact attention algorithm that dramatically reduces the memory footprint and wall-clock time of attention computation — without any approximation. The key insight is that standard attention is memory-bandwidth bound, not compute-bound: the bottleneck is reading and writing the large N×N attention matrix to GPU high-bandwidth memory (HBM).

Flash Attention restructures the computation using tiling: instead of materializing the full attention matrix in HBM, it processes the attention in small blocks that fit in the GPU's fast on-chip SRAM (shared memory). Each tile computes a partial softmax, and the results are combined using an online softmax algorithm that maintains numerical correctness without ever storing the full matrix.

The result: Flash Attention reduces memory usage from O(N²) to O(N) and achieves 2–4× wall-clock speedup over standard PyTorch attention, because it minimizes the number of reads/writes to slow HBM. Flash Attention 2 further optimizes the algorithm with better work partitioning across GPU warps and thread blocks.

⚡ How It Differs from Standard Attention

  • Standard: Computes full N×N attention matrix in HBM → O(N²) memory, many HBM reads/writes.
  • Flash Attention: Tiles the computation into SRAM-sized blocks → O(N) memory, minimal HBM traffic.
  • Exact computation — not an approximation. Produces bit-identical results (up to floating-point ordering).
  • 2–4× faster wall-clock time due to reduced memory bandwidth usage.
  • Enables much longer context lengths (since the N×N matrix is never materialized).
  • Fuses the entire attention operation (QKT, scaling, masking, softmax, dropout, V multiply) into a single kernel.

Key Points

  • IO-aware: The algorithm is designed around the GPU memory hierarchy (SRAM vs HBM), not just FLOPs.
  • Exact, not approximate: Unlike sparse or linear attention variants, Flash Attention computes the exact same result.
  • Universally adopted: Used by virtually all modern LLM training and inference systems (PyTorch 2.0+ includes it as scaled_dot_product_attention).
  • Enables long context: By avoiding the O(N²) memory, models can train on 32K–128K+ token sequences.
  • Flash Attention 2/3: Subsequent versions improve parallelism across warps and add features like variable-length sequences and FP8 support.
