Why GPU Tensor Cores Sit Idle During LLM Decode

An interactive visualization of the memory bandwidth bottleneck

The Problem in One Sentence

A modern H100 GPU can do ~2 petaflops of BF16 matrix math per second, yet during single-batch LLM decode, its Tensor Cores sit idle more than 99% of the time. The bottleneck isn't compute — it's HBM bandwidth.

Two related effects are at play:
  1. HBM bandwidth limit (hard ceiling). At batch=1, each decoded token must re-read all ~134 MB of weights through a single 3.35 TB/s HBM pipe. That sets a floor of ~40 µs per layer regardless of how many warps you launch — more warps can't invent bandwidth that isn't there.
  2. Latency hiding (small batches). Once a little parallel work exists (a few warps or a few in-flight requests), warps can overlap: while one waits for HBM, another issues MMA on a Tensor Core. This only helps up to the bandwidth ceiling above.

So the question "can't more warps fill the compute?" has a nuanced answer: yes, up to a point. Warps on the same SM don't fight for the single Tensor Core (an H100 SM actually has four, one per sub-partition), but they do share the SM's memory pipes and the chip's HBM bandwidth. Past the bandwidth ceiling, adding warps just makes them queue for HBM.

2. Why More Warps Can't Save Decode

A natural reaction is: "if one warp stalls, can't 48 warps just take turns and keep the Tensor Cores full?" In training (large batches) that's exactly what happens. In decode (batch=1), it doesn't — and not because of scheduling, but because of physics:

  • Weight bytes are fixed. To produce one token, the SM collectively must read the entire weight matrix from HBM. That's a fixed number of bytes, and HBM moves bytes at a fixed rate. More warps sharing the read doesn't reduce the total.
  • Warps compete for the same memory pipes. An SM has a small number of outstanding-load slots (MSHRs). Warps beyond that limit sit in a queue (shown in dark red) until a slot frees up. Adding warps past this point doesn't add bandwidth.
  • There's almost no reuse to amortize. Each weight is used in a single multiply-accumulate before being discarded. Arithmetic intensity is roughly the batch size. At batch=1, you do 1 FLOP per byte read — miles below the H100's ridge point of ~591 FLOP/byte.

So yes, warps do fill in for each other — but only up to what the memory subsystem can deliver. At batch=1 that ceiling is roughly HBM_bandwidth × cycle_time per SM, which is tiny relative to Tensor Core throughput.

6. Mitigations

Several strategies exist, all of which either raise arithmetic intensity or reduce HBM pressure.

Continuous Batching

Bundle multiple in-flight requests so more Warps share the SM. More Warps → more chances to hide latency.

Quantization (INT8 / FP4)

Cut weight size by 2-4× so each token requires proportionally less HBM traffic.

Speculative Decoding

A small draft model proposes several tokens; the large model verifies them in a single batched step.

More HBM Bandwidth

HBM3e on B200 roughly doubles bandwidth to ~8 TB/s, directly raising the memory roofline.

GQA / MQA

Share K/V heads across query heads. Smaller KV cache → less to read every decode step.

Mixture-of-Experts

Activate only a subset of weights per token, reducing the amount that must cross HBM.
