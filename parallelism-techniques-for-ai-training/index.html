1. Data Parallelism (DP)

Concept

Data Parallelism is the simplest and most widely used parallelism strategy. The entire model is replicated on every device (GPU). The global mini-batch is split into smaller micro-batches, one per device. Each device performs a full forward and backward pass on its micro-batch independently. After the backward pass, gradients are all-reduced (averaged) across all devices so every replica applies the same parameter update, keeping the models in sync.

Frameworks like PyTorch's DistributedDataParallel (DDP) implement this pattern. Because each device holds a full copy of the model, DP is limited by the memory capacity of a single device — the model must fit entirely in one GPU's memory.

When to Use

  • The model fits comfortably in a single GPU's memory.
  • You want to scale training throughput by adding more GPUs.
  • You need the simplest possible distributed setup.

⚠️ Limitations

  • Memory ceiling: Every GPU must hold the full model weights, optimizer states, and activations.
  • Communication overhead: All-reduce of gradients can become a bottleneck at large scale.
  • Batch-size scaling: Effective batch size grows linearly with GPU count, which can hurt convergence if not managed (learning-rate warmup, LARS/LAMB).

2. Model Parallelism (MP)

Concept

Model Parallelism is the umbrella term for splitting the model itself across multiple devices, rather than replicating it. When a model is too large to fit in a single GPU, you partition its layers or parameters across devices.

In the simplest form — sometimes called naïve model parallelism — consecutive layers are placed on different GPUs. During the forward pass, activations are sent from one GPU to the next. During the backward pass, gradients flow in reverse. This creates a sequential dependency: while one GPU computes, the others sit idle, leading to low utilization (the "bubble" problem).

Tensor Parallelism and Pipeline Parallelism (covered next) are refined forms of model parallelism that address this inefficiency.

When to Use

  • The model does not fit in a single GPU's memory.
  • As a conceptual starting point before applying TP or PP.

⚠️ Limitations

  • GPU under-utilization: Sequential dependency means only one GPU is active at a time in the naïve case.
  • High inter-device communication: Activations and gradients must be transferred between GPUs at every layer boundary.
  • Complex partitioning: Balancing compute across devices requires careful layer assignment.

3. Tensor Parallelism (TP)

Concept

Tensor Parallelism splits individual layers (tensors) across multiple devices. Instead of placing whole layers on different GPUs, TP partitions the weight matrices of a single layer — typically the large linear projections inside Transformer blocks.

For example, a linear layer Y = XW can be split column-wise: each GPU holds a slice of W and computes a partial result. The partial results are then combined via an all-reduce or all-gather operation. Megatron-LM popularized this approach for Transformer models, splitting the attention heads and MLP columns across GPUs within a single node.

Because TP requires frequent communication within each layer's forward and backward pass, it works best over high-bandwidth interconnects (e.g., NVLink within a node).

When to Use

  • Individual layers are too large for a single GPU.
  • GPUs are connected via high-bandwidth links (NVLink, NVSwitch).
  • You want to reduce memory per GPU without pipeline bubbles.

⚠️ Limitations

  • Bandwidth-sensitive: Requires NVLink-class interconnect; poor performance over PCIe or network.
  • Scaling limit: Typically limited to 4–8 GPUs within a single node.
  • Implementation complexity: Requires modifying the model code to split specific layers.

4. Pipeline Parallelism (PP)

Concept

Pipeline Parallelism partitions the model into stages, each placed on a different device, and then splits the mini-batch into smaller micro-batches. Micro-batches are fed into the pipeline in sequence so that multiple stages can be active simultaneously — much like an assembly line.

Schedules like GPipe and 1F1B (one-forward-one-backward) minimize the idle "bubble" at the start and end of each batch. In GPipe, all micro-batches do their forward passes first, then all backward passes. In 1F1B, forward and backward passes are interleaved to keep the pipeline fuller.

PP communicates only at stage boundaries (point-to-point sends), making it suitable for inter-node communication where bandwidth is lower than intra-node.

When to Use

  • Training across multiple nodes where inter-node bandwidth is limited.
  • The model has many sequential layers that can be evenly partitioned.
  • Combined with TP (intra-node) for 3D parallelism.

⚠️ Limitations

  • Pipeline bubble: Some idle time is unavoidable at the start/end of each batch.
  • Memory imbalance: First and last stages may hold more activations.
  • Micro-batch tuning: Too few micro-batches → large bubble; too many → high memory for in-flight activations.

5. FSDP / ZeRO

Concept

Fully Sharded Data Parallelism (FSDP) — based on Microsoft's ZeRO (Zero Redundancy Optimizer) — eliminates the memory redundancy of standard Data Parallelism. Instead of replicating the full model on every GPU, FSDP shards the model parameters, gradients, and optimizer states across all devices.

ZeRO defines three stages of sharding:

  • Stage 1: Shard optimizer states only (e.g., Adam momentum and variance).
  • Stage 2: Shard optimizer states + gradients.
  • Stage 3: Shard optimizer states + gradients + parameters (full sharding).

Before each layer's forward or backward computation, the full parameters for that layer are gathered from all devices (all-gather). After the backward pass, gradients are scattered back (reduce-scatter). This trades extra communication for dramatically lower per-GPU memory, enabling training of models that would otherwise not fit.

When to Use

  • Model fits in aggregate GPU memory but not in a single GPU.
  • You want to keep the simplicity of data-parallel training while reducing memory.
  • Good default choice for large model training before adding TP/PP.

⚠️ Limitations

  • Communication volume: Full sharding (Stage 3) increases all-gather/reduce-scatter traffic significantly.
  • Latency sensitivity: Frequent collective operations can hurt throughput on slow interconnects.
  • Complexity at scale: Tuning sharding groups, prefetching, and mixed precision requires expertise.

6. Expert Parallelism

Concept

Expert Parallelism is specific to Mixture-of-Experts (MoE) architectures. In an MoE model, certain layers (typically the feed-forward network in a Transformer block) are replaced with multiple "expert" sub-networks. A gating network (router) selects which experts process each token — usually the top-k experts.

With Expert Parallelism, different experts are placed on different devices. Tokens are routed to the appropriate device via all-to-all communication. This allows the model to have a very large total parameter count while keeping the per-token compute (FLOPs) manageable, since only a subset of experts is activated for each token.

Models like GShard, Switch Transformer, and Mixtral use this approach to scale to trillions of parameters.

When to Use

  • Training MoE architectures (Switch Transformer, Mixtral, etc.).
  • You want to scale parameter count without proportionally scaling compute.
  • Combined with DP and TP for large-scale MoE training.

⚠️ Limitations

  • All-to-all communication: Token routing requires expensive all-to-all collectives.
  • Load imbalance: If the router sends too many tokens to one expert, that GPU becomes a bottleneck.
  • Auxiliary losses: Requires load-balancing losses to prevent expert collapse.
  • Architecture-specific: Only applicable to MoE models.

7. Sequence Parallelism

Concept

Sequence Parallelism splits the sequence dimension of the input across devices. In a Transformer, operations like LayerNorm and Dropout operate independently on each token position. When combined with Tensor Parallelism, these non-tensor-parallel regions can be partitioned along the sequence dimension to further reduce activation memory.

There are two main flavors:

  • Megatron-style SP: Used alongside TP. The sequence is split for LayerNorm/Dropout regions, and all-gather/reduce-scatter operations convert between sequence-split and tensor-split representations at layer boundaries.
  • Ring Attention / Context Parallelism: Splits the full sequence across devices for the attention computation itself, passing KV blocks in a ring pattern. This enables training on extremely long sequences (100K+ tokens).

When to Use

  • Training with very long sequences (32K, 100K+ tokens).
  • Activation memory is the bottleneck (common with long contexts).
  • Already using TP and want to further reduce memory in non-TP regions.

⚠️ Limitations

  • Attention complexity: Ring attention adds communication rounds proportional to the number of devices.
  • Tightly coupled with TP: Megatron-style SP only makes sense when TP is already in use.
  • Implementation maturity: Less battle-tested than DP or TP in many frameworks.

Decision Guide: When to Choose & Combine

In practice, large-scale training uses a combination of these techniques — often called 3D parallelism (DP × TP × PP) or even 5D parallelism when adding FSDP and Sequence Parallelism. The table below helps you decide which techniques to apply based on your scenario.

Scenario Recommended Techniques Why
Model fits in 1 GPU, want faster training DP Simplest approach — replicate model, split data. Linear throughput scaling.
Model slightly exceeds 1 GPU memory FSDP/ZeRO Shard optimizer states and parameters across GPUs. Minimal code changes over DDP.
Large model, single-node (4–8 GPUs with NVLink) TP + DP TP splits layers within the node (fast NVLink); DP across nodes for throughput.
Very large model, multi-node TP + PP + DP 3D parallelism: TP intra-node, PP inter-node (low bandwidth OK), DP for throughput.
Very large model, want simplicity FSDP/ZeRO-3 Full sharding avoids manual TP/PP partitioning. Good up to moderate scale.
Extremely large model (100B+ params) TP + PP + FSDP + SP Combine all techniques. FSDP replaces pure DP for memory savings. SP for activations.
MoE architecture EP + DP + TP EP distributes experts; DP/TP handle the non-expert layers.
Long sequences (32K–100K+ tokens) SP + TP SP (Ring Attention) splits sequence to fit in memory; TP for layer-level splitting.
Memory-constrained, moderate model FSDP/ZeRO-2 + DP Stage 2 shards gradients + optimizer states with less communication than Stage 3.
Maximum scale (GPT-4 / Llama-3 class) TP + PP + DP + SP + EP 5D parallelism. Every dimension is exploited. Requires expert-level tuning.

Key Takeaways

  • Start simple: Use DP or FSDP first. Add TP/PP only when needed.
  • TP intra-node, PP inter-node: Match parallelism to your interconnect topology.
  • FSDP is the new DP: For models that don't fit in one GPU, FSDP/ZeRO is the easiest first step.
  • Combine incrementally: Add one dimension of parallelism at a time and measure throughput.
  • Profile everything: Use tools like PyTorch Profiler, NVIDIA Nsight, or DeepSpeed Flops Profiler to find bottlenecks.
