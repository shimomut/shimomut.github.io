Why GPU Tensor Cores Sit Idle During LLM Decode

An interactive visualization of the memory bandwidth bottleneck

The Problem in One Sentence

A modern H100 GPU can do ~2 petaflops of BF16 matrix math per second, yet during single-batch LLM decode, its Tensor Cores sit idle more than 99% of the time. The bottleneck isn't compute — it's HBM bandwidth.

Core insight: During training, many Warps coexist on an SM, so while some wait for HBM data, others keep the Tensor Cores busy. During decode (batch=1), there are too few Warps to hide the ~500-cycle HBM latency, so Tensor Cores are idle >99% of the time.

3. Roofline Model (H100)

The Roofline plots attainable performance against arithmetic intensity. Below the ridge point, you're memory-bound; above it, you're compute-bound. Decode lives deep in the memory-bound region. Training lives on the compute ceiling.

5. Mitigations

Several strategies exist, all of which either raise arithmetic intensity or reduce HBM pressure.

Continuous Batching

Bundle multiple in-flight requests so more Warps share the SM. More Warps → more chances to hide latency.

Quantization (INT8 / FP4)

Cut weight size by 2-4× so each token requires proportionally less HBM traffic.

Speculative Decoding

A small draft model proposes several tokens; the large model verifies them in a single batched step.

More HBM Bandwidth

HBM3e on B200 roughly doubles bandwidth to ~8 TB/s, directly raising the memory roofline.

GQA / MQA

Share K/V heads across query heads. Smaller KV cache → less to read every decode step.

Mixture-of-Experts

Activate only a subset of weights per token, reducing the amount that must cross HBM.
