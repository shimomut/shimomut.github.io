1. Simple time() Profiling

Python Built-in

Overview

The simplest profiling technique: wrap code regions with time.time() or time.perf_counter() calls and compute the elapsed duration. Zero dependencies, zero setup. It's the first thing you reach for when something feels slow and you want to narrow down which part of your pipeline is the culprit.

For GPU workloads, remember that CUDA operations are asynchronous — you must call torch.cuda.synchronize() before taking a timestamp, otherwise you're measuring kernel launch time, not kernel execution time.

When to Use

  • Quick sanity checks — "is data loading or the forward pass the bottleneck?"
  • Comparing wall-clock time of two implementations.
  • Measuring end-to-end pipeline latency (preprocessing → inference → postprocessing).
  • When you don't want to install any additional tools.

What You Can Measure

  • Wall-clock elapsed time for any code region.
  • Per-iteration training step time.
  • Data loading vs. compute time breakdown.
  • Latency of individual inference requests.

How to Use

CPU-only timing:

import time

start = time.perf_counter()
# ... your code ...
elapsed = time.perf_counter() - start
print(f"Elapsed: {elapsed:.3f}s")

GPU-aware timing (critical for CUDA workloads):

import torch, time

torch.cuda.synchronize()
start = time.perf_counter()

output = model(input_batch)
loss = criterion(output, labels)
loss.backward()

torch.cuda.synchronize()
elapsed = time.perf_counter() - start
print(f"Step time: {elapsed:.3f}s")

For repeated measurements, use PyTorch's built-in torch.utils.benchmark.Timer which handles warmup, synchronization, and statistical aggregation automatically.

2. PyTorch Profiler

Python GPU

Overview

PyTorch Profiler (torch.profiler) is the framework-native profiling tool. It captures CPU operator execution, CUDA kernel launches, GPU kernel execution, memory allocations, and communication operations (NCCL collectives) — all correlated on a unified timeline. Results can be viewed in TensorBoard via the PyTorch Profiler plugin or exported as Chrome traces.

The profiler wraps your training loop with a context manager and uses a schedule to control when profiling is active (skip warmup iterations, profile a few steps, then stop). This keeps overhead manageable while capturing representative data.

When to Use

  • Identifying which operators or kernels dominate training step time.
  • Finding CPU-GPU synchronization stalls (e.g., accidental .item() calls).
  • Analyzing NCCL collective durations in distributed training.
  • Detecting memory allocation patterns and peak memory usage.
  • Debugging OOM errors — finding which tensors consume the most memory.
  • Tuning batch size to maximize GPU memory utilization without OOM.
  • Verifying that gradient checkpointing or mixed precision actually reduced memory.
  • Tracking memory fragmentation in long-running training jobs.
  • Comparing performance before/after an optimization (e.g., torch.compile, mixed precision).

What You Can Measure

  • Per-operator CPU and CUDA time (self time vs. total time).
  • CUDA kernel execution time and occupancy.
  • GPU memory allocation/deallocation timeline.
  • Peak and current GPU memory allocated / reserved.
  • Per-tensor memory attribution (with memory snapshots).
  • Memory fragmentation and allocator efficiency.
  • NCCL collective operation durations (AllReduce, AllGather, etc.).
  • Data loader worker time and CPU-side preprocessing.
  • Tensor shapes flowing through each operator.
  • GPU SM utilization, power draw, temperature, and clock speeds (via nvidia-smi).

How to Use

from torch.profiler import profile, schedule, tensorboard_trace_handler

with profile(
    activities=[
        torch.profiler.ProfilerActivity.CPU,
        torch.profiler.ProfilerActivity.CUDA,
    ],
    schedule=schedule(wait=1, warmup=1, active=3, repeat=1),
    on_trace_ready=tensorboard_trace_handler("./log/profiler"),
    record_shapes=True,
    profile_memory=True,
    with_stack=True,
) as prof:
    for step, batch in enumerate(dataloader):
        train_step(batch)
        prof.step()

# Print top operators by CUDA time
print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=20))

Then launch TensorBoard to explore the trace: 

tensorboard --logdir=./log/profiler

GPU Memory Profiling

Understanding GPU memory consumption is critical for AI workloads. During training, GPU memory holds model parameters, optimizer states, gradients, activations (for the backward pass), and temporary buffers. A single misconfiguration — wrong batch size, missing gradient checkpointing, or an unexpected tensor copy — can cause an OOM crash or leave expensive GPU memory underutilized.

The profiler's profile_memory=True flag (shown above) captures the full allocation timeline. In addition, PyTorch provides dedicated memory analysis APIs: torch.cuda.memory_stats() for detailed allocator statistics, torch.cuda.memory_summary() for a human-readable snapshot, and torch.cuda.memory._record_memory_history() for capturing allocation traces that can be visualized in a flame graph.

Quick memory check in code:

import torch

# Current and peak memory
print(f"Allocated: {torch.cuda.memory_allocated() / 1e9:.2f} GB")
print(f"Reserved:  {torch.cuda.memory_reserved() / 1e9:.2f} GB")
print(f"Peak:      {torch.cuda.max_memory_allocated() / 1e9:.2f} GB")

# Detailed summary
print(torch.cuda.memory_summary())

Memory snapshot with allocation traces (PyTorch 2.1+):

# Start recording
torch.cuda.memory._record_memory_history(max_entries=100000)

# ... run your training step ...

# Save snapshot for visualization
torch.cuda.memory._dump_snapshot("memory_snapshot.pickle")
torch.cuda.memory._record_memory_history(enabled=None)

Upload the snapshot to pytorch.org/memory_viz for an interactive flame graph showing exactly which call sites allocated memory.

System-level monitoring with nvidia-smi:

# One-shot status
nvidia-smi

# Continuous monitoring every 1 second
nvidia-smi dmon -s um -d 1

# Query specific metrics as CSV
nvidia-smi --query-gpu=timestamp,gpu_name,utilization.gpu,memory.used,memory.total,power.draw --format=csv -l 1

3. Nsight Systems & Nsight Compute

GPU System-level

Overview

NVIDIA's Nsight tools are the gold standard for deep GPU performance analysis. They operate at two different levels of granularity:

Nsight Systems provides a system-wide timeline view — CPU threads, CUDA API calls, GPU kernels, NCCL operations, memory transfers, and OS runtime events all on a single correlated timeline. Think of it as a "big picture" tool: it shows you where time is spent and where the gaps are (CPU idle while GPU works, GPU idle waiting for data, NCCL stalls, etc.).

Nsight Compute is a kernel-level profiler. Once Nsight Systems tells you which kernel is the bottleneck, Nsight Compute tells you why. It collects hardware performance counters — SM occupancy, memory throughput, warp stall reasons, instruction mix, L1/L2 cache hit rates — and compares them against the theoretical peak of your GPU (the "roofline" model).

When to Use

  • Nsight Systems: Understanding the overall execution timeline — finding gaps, stalls, and overlap between CPU/GPU/network.
  • Nsight Systems: Profiling distributed training to see NCCL collectives relative to compute.
  • Nsight Systems: Verifying that CUDA Graphs, async data loading, or prefetching are working as expected.
  • Nsight Compute: Deep-diving into a specific CUDA kernel that dominates runtime.
  • Nsight Compute: Determining if a kernel is compute-bound or memory-bound (roofline analysis).
  • Nsight Compute: Optimizing custom CUDA/Triton kernels.

What You Can Measure

Nsight Systems:

  • End-to-end timeline of CPU, GPU, and network activity.
  • CUDA API call durations and kernel launch latency.
  • NCCL collective operation timing and overlap with compute.
  • CPU thread activity and OS scheduling events.
  • Memory copy (HtoD, DtoH, DtoD) durations and throughput.

Nsight Compute:

  • SM occupancy and warp scheduling efficiency.
  • Arithmetic throughput vs. memory throughput (roofline position).
  • L1/L2 cache hit rates and memory bandwidth utilization.
  • Warp stall reasons (memory dependency, execution dependency, etc.).
  • Instruction mix (FP16, FP32, INT, Tensor Core utilization).

How to Use

Nsight Systems — collect a trace from the command line:

# Profile a training script, capture CUDA + NCCL + OS runtime
nsys profile \
  --trace=cuda,nvtx,osrt,cudnn,cublas \
  --output=training_trace \
  --force-overwrite=true \
  python train.py

# For distributed training (profile rank 0 only)
nsys profile \
  --trace=cuda,nvtx,osrt \
  --cuda-memory-usage=true \
  --output=rank0_trace \
  torchrun --nproc_per_node=8 train.py

Open the resulting .nsys-rep file in the Nsight Systems GUI to explore the timeline. Use NVTX annotations in your code to mark regions of interest: 

import torch

with torch.cuda.nvtx.range("forward_pass"):
    output = model(input_batch)

with torch.cuda.nvtx.range("backward_pass"):
    loss.backward()

Nsight Compute — profile a specific kernel:

# Profile all kernels (slow — use for short runs)
ncu --set full -o kernel_report python inference.py

# Profile only kernels matching a name pattern
ncu --kernel-name "ampere_fp16_s1688gemm" \
    --launch-skip 5 --launch-count 3 \
    -o gemm_report \
    python inference.py

4. py-spy

Python System-level

Overview

py-spy is a sampling profiler for Python programs. Unlike cProfile (which instruments every function call and adds significant overhead), py-spy works by periodically sampling the call stack of a running Python process from outside — it attaches to the process without modifying it or requiring any code changes.

This makes py-spy uniquely useful for AI workloads: you can attach it to a long-running training job that's already in progress, see where CPU time is being spent, and detach — all without restarting the job or adding instrumentation. It's especially valuable for diagnosing Python-side bottlenecks: slow data preprocessing, GIL contention, inefficient tokenization, or unexpected Python overhead in the training loop.

When to Use

  • Profiling a running process without restarting it (attach to PID).
  • Finding CPU-side bottlenecks in data loading, preprocessing, or tokenization.
  • Diagnosing GIL contention in multi-threaded Python code.
  • Quick "where is my Python code spending time?" without any code changes.
  • Profiling code you can't easily modify (third-party libraries, complex launchers).

What You Can Measure

  • Python function call frequency and time (sampled, not exact).
  • Call stack flame graphs showing the hot path.
  • GIL usage and contention across threads.
  • Native (C extension) vs. Python frame time.

How to Use

# Install
pip install py-spy

# Record a flame graph SVG from a running process
py-spy record --pid 12345 --output profile.svg --duration 30

# Record from a new process
py-spy record --output profile.svg -- python train.py

# Live top-like view of a running process
py-spy top --pid 12345

# Include native C/C++ frames (useful for seeing into PyTorch internals)
py-spy record --pid 12345 --native --output profile.svg

The generated SVG flame graph can be opened in any browser. The width of each bar represents the proportion of samples where that function was on the stack — wider bars mean more time spent there.

5. Prometheus & Grafana

Infrastructure System-level

Overview

Prometheus is a time-series metrics database that scrapes numeric metrics from instrumented targets at regular intervals. Grafana is the visualization layer that queries Prometheus and renders dashboards with graphs, heatmaps, and alerts.

Together, they form the standard observability stack for AI training clusters. While tools like PyTorch Profiler and Nsight give you deep, short-duration profiles of a single run, Prometheus/Grafana give you continuous, long-duration monitoring across your entire fleet — GPU utilization trends over days, memory usage patterns across hundreds of nodes, network throughput during distributed training, and alerting when metrics deviate from expected ranges.

Key exporters for AI workloads include DCGM Exporter (GPU metrics via NVIDIA DCGM), Node Exporter (CPU, memory, disk, network), and custom application-level metrics exposed via the Prometheus client library.

When to Use

  • Continuous monitoring of GPU utilization, memory, and temperature across a cluster.
  • Tracking training throughput (samples/sec, tokens/sec) over time.
  • Alerting on anomalies — GPU utilization drops, OOM events, node failures.
  • Correlating infrastructure metrics with training loss curves.
  • Capacity planning — understanding resource utilization patterns.
  • Post-mortem analysis of failed training runs.

What You Can Measure

  • GPU utilization, memory usage, power draw, temperature (via DCGM Exporter).
  • CPU utilization, system memory, disk I/O, network throughput (via Node Exporter).
  • Custom application metrics: training loss, learning rate, throughput, batch time.
  • NCCL/EFA network bandwidth and error rates.
  • Job scheduling and queue wait times (Slurm/Kubernetes metrics).

How to Use

Expose custom training metrics from your Python code:

from prometheus_client import start_http_server, Gauge, Summary

# Create metrics
gpu_util = Gauge('training_gpu_utilization', 'GPU utilization %', ['gpu_id'])
step_time = Summary('training_step_seconds', 'Time per training step')
throughput = Gauge('training_throughput_tokens_per_sec', 'Training throughput')

# Start metrics server on port 8000
start_http_server(8000)

for step, batch in enumerate(dataloader):
    with step_time.time():
        train_step(batch)
    throughput.set(tokens_per_second)
    gpu_util.labels(gpu_id='0').set(get_gpu_utilization())

Deploy DCGM Exporter for GPU metrics (Docker example):

docker run -d --gpus all --rm \
  -p 9400:9400 \
  nvcr.io/nvidia/k8s/dcgm-exporter:latest

Configure Prometheus to scrape these endpoints, then build Grafana dashboards. NVIDIA provides a pre-built DCGM Exporter dashboard for Grafana.

6. File System Metrics (FSx for Lustre)

Storage Infrastructure

Overview

AI training workloads are often I/O-intensive — loading large datasets, reading checkpoint files, and writing model snapshots. When the file system can't keep up, GPUs sit idle waiting for data. Amazon FSx for Lustre is a high-performance parallel file system commonly used for AI/ML training on AWS, and monitoring its performance metrics is essential for diagnosing data pipeline bottlenecks.

FSx for Lustre publishes metrics to Amazon CloudWatch, and Lustre itself exposes detailed statistics via /proc and lctl commands on the client nodes. Key concerns include: aggregate throughput (are you hitting the file system's provisioned bandwidth?), metadata IOPS (are small-file operations like open()/stat() creating a bottleneck?), and client-side caching behavior.

When to Use

  • GPU utilization is low and you suspect the data pipeline is the bottleneck.
  • Training throughput drops when scaling to more nodes (I/O contention).
  • Checkpoint save/load times are unexpectedly long.
  • You need to right-size your FSx for Lustre file system (throughput tier, storage capacity).
  • Diagnosing "thundering herd" problems where all nodes read the same files simultaneously.

What You Can Measure

  • Throughput: DataReadBytes, DataWriteBytes — aggregate read/write bandwidth.
  • IOPS: DataReadOperations, DataWriteOperations — I/O operations per second.
  • Metadata IOPS: MetadataOperations — file open, stat, readdir, create operations.
  • Free storage capacity: FreeDataStorageCapacity — remaining disk space.
  • Client-side stats: Lustre OSC/MDC statistics (read/write bytes, RPC counts, cache hits).
  • Latency: Per-operation latency distributions from Lustre client stats.

How to Use

CloudWatch metrics (via AWS CLI):

# Get read throughput for an FSx file system over the last hour
aws cloudwatch get-metric-statistics \
  --namespace AWS/FSx \
  --metric-name DataReadBytes \
  --dimensions Name=FileSystemId,Value=fs-0123456789abcdef0 \
  --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%S) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%S) \
  --period 60 \
  --statistics Sum

Client-side Lustre statistics (run on compute nodes):

# Overall client stats
lctl get_param llite.*.stats

# OST (Object Storage Target) read/write stats
lctl get_param osc.*.stats

# MDC (Metadata Client) stats — shows metadata operation counts
lctl get_param mdc.*.stats

# Real-time monitoring with watch
watch -n 1 'lctl get_param llite.*.stats | grep -E "read_bytes|write_bytes"'

Quick I/O benchmark to validate file system performance:

# Sequential write throughput test (1GB file)
dd if=/dev/zero of=/fsx/benchmark_test bs=1M count=1024 oflag=direct

# Sequential read throughput test
dd if=/fsx/benchmark_test of=/dev/null bs=1M iflag=direct

# Parallel I/O test with fio
fio --name=seqread --rw=read --bs=1M --size=1G \
    --numjobs=8 --directory=/fsx/bench --direct=1

Comparison & Decision Guide

Each tool operates at a different level of the stack and answers different questions. The table below helps you pick the right tool for your situation.

Tool Level Best For Overhead Requires Code Changes
time() / perf_counter Application Quick wall-clock measurements Negligible Yes (minimal)
PyTorch Profiler Framework Operator-level CPU/GPU analysis, memory debugging, OOM analysis Low–Medium Yes (context manager)
Nsight Systems System Full timeline: CPU + GPU + network Low–Medium No (optional NVTX)
Nsight Compute Kernel Deep kernel analysis, roofline High (replays kernels) No
py-spy Application (CPU) Python CPU profiling, GIL analysis Very low No (attaches to PID)
Prometheus / Grafana Infrastructure Continuous cluster monitoring, alerting Negligible Setup required
FSx Lustre metrics Storage I/O throughput, metadata IOPS Negligible No

Recommended Workflow

  • Start broad, then narrow down. Use time() to identify which phase is slow, then use specialized tools to dig deeper.
  • GPU underutilized? Check Nsight Systems for CPU-GPU gaps. Check py-spy for Python-side bottlenecks. Check FSx metrics for I/O stalls.
  • GPU fully utilized but slow? Use PyTorch Profiler to find the expensive operators, then Nsight Compute to optimize the hot kernels.
  • OOM errors? Use torch.cuda.memory_summary() and memory snapshots to find the culprit tensors.
  • Scaling problems? Use Nsight Systems to check NCCL collective overhead. Use Prometheus/Grafana to monitor per-node GPU utilization across the cluster.
  • Always have Prometheus/Grafana running on your training cluster — it's your continuous baseline that catches regressions and anomalies you wouldn't notice otherwise.
