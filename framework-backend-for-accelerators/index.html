Framework Backends for Non-GPU Accelerators

How PyTorch, JAX, and inference frameworks adapt their backends for TPU, Trainium, and other accelerators

Modern ML frameworks are designed with a pluggable backend architecture. The user-facing API (e.g., torch.nn, model.generate()) stays the same, but the compilation and execution layers underneath can be swapped to target different hardware.

This article visualizes the software stack for inference and training across accelerators, highlighting which layers must be customized and which remain shared.

How torch.compile Enables Hardware Portability

torch.compile (PyTorch 2.0+) introduces a clean separation between graph capture and graph lowering. TorchDynamo captures the computation graph from Python, then hands it to a pluggable backend for optimization and code generation.

For NVIDIA GPUs, the default backend is TorchInductor, which generates Triton kernels. For Trainium, AWS replaces this with torch_neuronx (backed by Neuron Compiler), which lowers the same graph to NeuronCore-optimized instructions. For TPUs, Google provides torch_xla, which routes through XLA/HLO.

Important: torch.compile is primarily the inference portability story

On NVIDIA GPUs, torch.compile works for both training and inference. However, on non-GPU accelerators, the torch.compile backend-replacement pattern is primarily used for inference.

For training, non-GPU accelerators typically use a different mechanism — XLA lazy tensors. Instead of tracing via TorchDynamo, the model runs in eager mode but operations are accumulated into an XLA graph (lazily), which is then compiled and executed when a synchronization point is reached. This approach better handles the dynamic nature of training (gradient computation, optimizer steps, distributed communication).

Key Design Patterns

Graph Capture (Shared)

TorchDynamo / torch.fx traces the Python model into an intermediate representation (FX Graph). This layer is hardware-agnostic and shared across all backends.

Graph Lowering (Custom)

Each accelerator provides its own compiler that transforms the FX Graph into hardware-specific instructions. This is where operator fusion, memory planning, and scheduling happen.

Runtime & Driver (Custom)

The compiled artifact executes on a device-specific runtime. CUDA Runtime for GPUs, Neuron Runtime for Trainium, XRT for TPUs. Each manages device memory, kernel dispatch, and multi-device communication.

Collective Communication (Custom)

Distributed operations (AllReduce, AllGather) use hardware-specific libraries: NCCL for GPUs, Neuron CCL for Trainium NeuronLink, and XLA SPMD for TPU ICI.

Inference vs Training: What Changes?

For inference, the torch.compile backend-replacement pattern provides a clean portability story. The key customization points are the compiler backend and the serving runtime. Inference-specific optimizations (KV cache management, continuous batching, speculative decoding) need framework-level adaptation for each accelerator.

For training, the graph capture mechanism itself changes on non-GPU accelerators. Instead of TorchDynamo, training relies on XLA lazy tensors or framework-native tracing (JAX's jit). Additional customization is needed for gradient computation, distributed communication (collective operations), and mixed-precision support (hardware-specific numeric formats like NeuronCore's stochastic rounding or TPU's native bfloat16).

This means the "plug in a different backend" abstraction is thinner for training — you often need deeper integration with the framework's autograd, optimizer, and distributed modules, which is why each accelerator vendor provides a training-specific SDK (NxD, torch_xla, Synapse AI).
